Why Pipenv?

Pipenv is a tool that aims to bring the best of all packaging worlds (bundler, composer, npm, cargo, yarn, etc.) to the Python world. It automatically creates and manages a virtualenv for your projects, as well as adds/removes packages from your Pipfile as you install/uninstall packages. It also generates the ever-important Pipfile.lock, which is used to replicate same environment in the production.

The problems that Pipenv seeks to solve are multi-faceted:

  • We no longer need to use pip and virtualenv separately. They work together.
  • No need to manage a requirements.txt file, Pipfile and Pipfile.lock  is superior for basic use cases.
  • Hashes are used everywhere, always. Security.
  • Give you insight into your dependency graph (e.g. $ pipenv graph).
  • Streamline development workflow by loading .env files.

pipenv Installation & Configuration

Ubuntu
$sudo apt install pipenv
Windows
>pip install pipenv

Executing “pipenv shell” command creates a virtual environment in the preconfigured pipenv location. Usually this location is a hidden folder in user’s home directory.
How pipenv decide the name of the virtual environment dynamically?
pipenv read the name of the project folder from where we are executing the command pipenv. Then it creates the hash using the Project folder name. pipenv use this hashname to determine the virtualenv name. So whenever we execute “pipenv shell” from any project folder and pipenv generate hash name of the project folder and find the correct virtual environment from the list of available virtual environments, and then activate it.
In following example all virtual environments are in “/home/kaushik/.local/share/virtualenvs/” folder
temp1-z5LkJ42k temp2-8-i5HDWi hash name created using folder name.
Problem: If we change the folder/project name in future and run pipenv shell, pipenv will not be able to locate the existing virtual environment for that project as hash name will get change. Instead pipenv will create a new virtual environment using the new project name.
I want pipenv to create the virtual environment in the same project folder rather than creating it in a common location.
Run following command before running “pipenv shell” from any project folder. 

Ubuntu
$ export PIPENV_VENV_IN_PROJECT=true

What happens when I installed a package called “requests”
It update the existing Pipfile and create on more file called Pipfile.lock
In Pipfile
requests = “*”
“*” –> Asterisk means as I have not specified any specific version of requests so it will consider the latest version.
In Pipfile.lock
“version”: “==2.23.0”
Here we can find the exact requests version got install and all it’s dependencies. It contains all hashes for packages.
After completion of our Project we use the Pipfile.lock file to replicate the same environment in the production.

Note: Pipfile is not deterministic but Pipfile.lock is.

Find Dependencies

Using “pipenv graph” we can find specific version of requests package depends on what other packages and what their version. In a large project version conflict between packages are very common. Say example pckg1 depends on package dpnds version==1.0 and pckg2 depends on package dpnds version==2.0. Usually in such cases we go for higher version of dpnds, expecting there will be backward compatibility. Many other ways we can resolve it

kaushik@kaushik-Inspiron-5547:~/temp2$ pipenv graph
requests==2.23.0
  - certifi [required: >=2017.4.17, installed: 2019.11.28]
  - chardet [required: >=3.0.2,<4, installed: 3.0.4]
  - idna [required: >=2.5,<3, installed: 2.9]
  - urllib3 [required: >=1.21.1,<1.26,!=1.25.1,!=1.25.0, installed: 1.25.8]

run commands within virtual environment without activating it

We can run a command within our virtual environment without activating the environment. Just by using “pipenv run“. Advantage here is no need to run exit command to come out of pipenv shell after execution.
pipenv use subshell to run our commands. So we don’t use activate command as we use to do it with virtualenv.

How to replicate virtual environment of an old project

Previously we use to use virtualenv, pip and pip freeze to create requirements.txt file. If we want to move to pipenv environment:
$pipenv install -r requirements.txt
We can create our own dependency file as well using pipenv lock -r

kaushik@kaushik-Inspiron-5547:~/temp2$ pipenv lock -r
-i https://pypi.org/simple

idna==2.9
requests==2.23.0
urllib3==1.25.8

The project folder will still have Pipfile and Pipfile.lock. To recreate the virtual environment just execute pipenv install.
It will read from Pipfile and Pipfile.lock to create the environment

Environment Variable

Set environment variable for a specific environment is extremely important. Say working on multiple flask and django projects. Every project has got it’s own environment variables like secret keys, database connection informations etc. Good if virtual environment can handle this without doing it manually. To do this create .env file in project folder. Keep all our environment variable in that file. From Python we can access those variables
Note: Remember don’t put your environment variable in git repositories. As it has many sensitive data. So put .env file in .gitignore file. 

kaushik@kaushik-Inspiron-5547:~/temp2$ cat >> .env
SECRET_KEY="LovePython"

kaushik@kaushik-Inspiron-5547:~/temp2$ pipenv run python

Usage Examples:
   Create a new project using Python 3.7, specifically:
   $ pipenv --python 3.7

   Remove project virtualenv (inferred from current directory):
   $ pipenv --rm

   Install all dependencies for a project (including dev):
   $ pipenv install --dev

   $ pipenv lock --pre

   $ pipenv graph

   Check your installed dependencies for security vulnerabilities:
   $ pipenv check
